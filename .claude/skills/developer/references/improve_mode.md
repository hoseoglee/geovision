# Improve 모드 상세 (/developer improve)

`/developer improve`는 현재 프로젝트의 우려 사항을 자동으로 탐색하고, 사용자에게 선택지를 제시한 뒤, 선택된 항목을 개발 프로세스(Phase 1~8)로 실행한다.

## Improve 워크플로우

```
┌──────────────────────────────────────────────────────────┐
│  Step 1: SCAN (병렬 — 관점별 Agent 탐색)                  │
│  ├─ Agent A: 코드 품질 — 중복, 복잡도, 미사용 코드, 타입  │
│  ├─ Agent B: 테스트 — 커버리지 갭, 엣지 케이스 미검증      │
│  ├─ Agent C: 보안 — 하드코딩 시크릿, 인젝션, 입력 검증     │
│  ├─ Agent D: 성능 — N+1 쿼리, 불필요한 I/O, 메모리 누수   │
│  └─ Agent E: 구조 — 폴더 구조 표준 위반, 로깅 규약 미적용  │
│                                                          │
│  Step 2: PRIORITIZE (순차 — 결과 종합)                    │
│  └─ 발견 사항을 심각도(critical/high/medium/low)로 분류    │
│     → 번호 매긴 제안 목록을 사용자에게 제시                │
│                                                          │
│  Step 3: SELECT (사용자 입력)                             │
│  └─ 사용자가 번호로 선택 (복수 선택 가능)                  │
│                                                          │
│  Step 4: EXECUTE (선택 항목별 개발 프로세스)               │
│  └─ 각 선택 항목을 Phase 1~8로 실행                       │
│     - 간단한 수정 → Phase 3(BUILD) + Phase 4(HARNESS)    │
│     - 구조 변경 → Phase 1(SPEC)부터 풀 프로세스           │
│     - 독립 항목은 Agent로 병렬 실행                       │
└──────────────────────────────────────────────────────────┘
```

## Step 1: SCAN 상세

`--scope` 옵션으로 탐색 범위를 제한할 수 있다. 기본값은 `all`.

```
동시에 실행 (최대 5개 Agent):
├─ Agent A: 코드 품질 (--scope code)
│   - 중복 코드 (함수/로직 유사도)
│   - 복잡한 함수 (분기 10개 이상, 100줄 이상)
│   - 미사용 임포트/변수/함수
│   - 타입 힌트 누락 (공개 API)
│   - 에러 처리 미흡 (bare except, 무시된 예외)
│
├─ Agent B: 테스트 (--scope test)
│   - 테스트 없는 모듈/함수
│   - 엣지 케이스 미검증 (빈 입력, None, 경계값)
│   - 모킹 과다 (실제 로직 미검증)
│   - 테스트 격리 부족 (순서 의존, 공유 상태)
│
├─ Agent C: 보안 (--scope security)
│   - 하드코딩된 시크릿/토큰
│   - SQL 인젝션, XSS, 커맨드 인젝션 가능성
│   - 입력 검증 미흡
│   - 의존성 취약점 (알려진 CVE)
│
├─ Agent D: 성능 (--scope performance)
│   - N+1 쿼리 패턴
│   - 불필요한 동기 I/O (파일, 네트워크)
│   - 대용량 데이터 메모리 로드
│   - 캐싱 미적용
│
└─ Agent E: 구조 (--scope all에서만)
    - 폴더 구조 표준 위반 (루트에 실행 코드 3개+)
    - 로깅 규약 미적용 (logs/error.log 부재)
    - .gitignore 누락 항목
    - README/문서 부재
```

**Agent 프롬프트 규칙:**
- 프로젝트 루트 경로, constitution.md, 기존 스펙/테스트 경로를 전달
- 각 Agent는 **발견 사항만 보고** (코드 수정하지 않음)
- 출력 형식: `[심각도] 파일:라인 — 설명 — 제안`

## Step 2: PRIORITIZE

Agent 결과를 종합하여 다음 형식으로 사용자에게 제시한다:

```
🔍 프로젝트 개선 제안 (총 N건)

🔴 Critical (즉시 수정)
  1. [보안] src/config.py:42 — API 토큰 하드코딩 → 환경변수로 이동

🟠 High (강력 권장)
  2. [테스트] src/worker.py — execute_task 타임아웃 엣지 케이스 미검증
  3. [구조] 루트에 실행 코드 15개 → src/로 이동 필요

🟡 Medium
  4. [코드] src/bot.py:44-71 — handle_message/handle_urgent 90% 중복 → 공통 함수 추출
  5. [성능] src/worker.py:88 — 매 실행마다 파일 전체 스캔 → 인덱스 캐싱

🟢 Low
  6. [코드] src/config.py — 타입 힌트 누락 (공개 함수 3개)

실행할 항목 번호를 선택하세요 (예: 1,2,4 또는 all):
```

## Step 3: SELECT

- 사용자가 번호를 선택하면 해당 항목만 실행
- `all`: 전체 실행 (critical부터 순서대로)
- `critical`: critical만 실행
- 선택하지 않으면 종료

## Step 4: EXECUTE

선택된 각 항목의 **규모에 따라 적절한 Phase부터 시작**한다:

| 규모 | 시작 Phase | 예시 |
|------|-----------|------|
| 소규모 (1~2파일 수정) | Phase 3 (BUILD) → Phase 4 (HARNESS) | 하드코딩 제거, 타입 힌트 추가 |
| 중규모 (모듈 리팩터링) | Phase 1 (SPEC) → 풀 프로세스 | 중복 함수 추출, 테스트 추가 |
| 대규모 (구조 변경) | Phase 1 (SPEC) → 풀 프로세스 | 폴더 구조 리팩터링, 아키텍처 변경 |

- **독립적인 소규모 항목**은 Agent로 병렬 실행
- **의존성 있는 항목**은 순차 실행 (예: 구조 변경 후 테스트 추가)
- 각 항목 완료 후 테스트 하네스(Phase 4) 실행하여 기존 테스트 깨지지 않음을 확인
