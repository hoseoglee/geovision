# Decompose Skill

큰 요청을 서브태스크로 분해하고 의존성을 분석하여 최적의 실행 계획을 수립한다.

## 워크플로우

### 1. 사용자 요청 분석
- 사용자의 요청을 파악한다.
- 요청이 이미 단순한 작업(단일 파일 수정, 단답형 질문 등)이면 분해 없이 바로 실행을 안내한다.
- 요청의 범위, 복잡도, 관련 시스템을 파악한다.

### 2. 서브태스크 분해
- 요청을 독립적이고 실행 가능한 서브태스크로 분해한다.
- 각 태스크에 고유 ID를 부여한다: `T-1`, `T-2`, `T-3`, ...
- 각 태스크에 다음 정보를 포함한다:
  - ID
  - 제목
  - 설명 (구체적으로 무엇을 하는지)
  - 예상 복잡도: 낮음 / 중간 / 높음
  - 예상 소요시간: 짧음(~1분) / 보통(~5분) / 길음(~10분+)

### 3. 의존성 분석
- 태스크 간 의존 관계를 분석한다.
- 의존성 유형:
  - **순차(sequential)**: T-2가 T-1의 결과를 필요로 하는 경우
  - **병렬(parallel)**: T-2와 T-3이 서로 독립적인 경우
  - **조건부(conditional)**: T-1의 결과에 따라 T-2 또는 T-3을 실행하는 경우

### 4. 의존성 그래프 시각화
- 텍스트 기반으로 의존성 그래프를 시각화한다:

```
T-1 (기반 설정)
├── T-2 (모듈 A 구현)  ──┐
│                         ├── T-5 (통합 테스트)
├── T-3 (모듈 B 구현)  ──┘
│
└── T-4 (문서 작성)  [병렬 가능]
```

- 실행 그룹을 명시한다:
  - **Phase 1** (순차): T-1
  - **Phase 2** (병렬): T-2, T-3, T-4
  - **Phase 3** (순차): T-5

### 5. 실행 계획 요약
- 전체 태스크 목록을 테이블로 보여준다:

```
| ID   | 태스크        | 의존성 | 복잡도 | 소요시간 | 실행 Phase |
|------|-------------|--------|--------|---------|-----------|
| T-1  | 기반 설정     | 없음   | 낮음   | ~1분    | Phase 1   |
| T-2  | 모듈 A 구현   | T-1   | 중간   | ~5분    | Phase 2   |
| T-3  | 모듈 B 구현   | T-1   | 중간   | ~5분    | Phase 2   |
| T-4  | 문서 작성     | T-1   | 낮음   | ~3분    | Phase 2   |
| T-5  | 통합 테스트   | T-2,3 | 높음   | ~10분   | Phase 3   |
```

- 총 예상 소요시간 (순차 실행 시 vs 병렬 실행 시) 을 표시한다.

### 6. 사용자 확인 후 실행
- 실행 계획을 사용자에게 보여주고 확인을 받는다.
- 사용자가 승인하면:
  - 각 Phase를 순서대로 실행한다.
  - Phase 내에서 병렬 가능한 태스크는 Agent 도구를 사용하여 병렬 실행한다.
  - 각 태스크 완료 시 상태를 업데이트한다.
- 사용자가 수정을 요청하면 계획을 조정한다.

## 핵심 원칙

- **적절한 세분화**: 너무 잘게 쪼개지 않는다 (1줄 수정이 하나의 태스크가 되면 안 됨). 의미 있는 단위로 분해한다.
- **의존성 정확성**: 의존성을 잘못 판단하면 병렬 실행 시 오류가 발생한다. 보수적으로 판단한다.
- **사용자 통제**: 실행 전 반드시 사용자 확인을 받는다. 자동 실행하지 않는다.
- **유연성**: 실행 중 예상치 못한 상황이 발생하면 계획을 재조정한다.
- **병렬 우선**: 병렬 실행 가능한 태스크는 적극적으로 병렬 처리하여 시간을 단축한다.

## 검증 기준

- [ ] 각 서브태스크에 병렬/순차 판단이 표시됨
- [ ] 실행 계획이 사용자에게 제시됨
