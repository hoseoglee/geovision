# Test Harness Skill

테스트를 실행하고, 실패하면 코드를 수정하고, 다시 테스트하는 Red-Green-Refactor 루프를 자동으로 수행한다.

## 사용법

```
/test_harness [test_command] [--max-iterations N]
```

- `test_command`: 테스트 실행 명령 (기본: 자동 감지)
- `--max-iterations`: 최대 반복 횟수 (기본: 10)

## 워크플로우

### 1. 테스트 환경 감지

프로젝트의 테스트 프레임워크를 자동 감지한다:

| 파일 | 프레임워크 | 명령어 |
|------|-----------|--------|
| `pytest.ini`, `pyproject.toml` | pytest | `pytest -v` |
| `package.json` (test script) | npm | `npm test` |
| `Cargo.toml` | cargo | `cargo test` |
| `go.mod` | go | `go test ./...` |
| `test_*.py` 파일 존재 | pytest | `pytest -v` |

사용자가 명령어를 지정하면 그것을 사용한다.

### 2. Red-Green-Refactor 루프

```
┌─────────────────────────────────────┐
│  1. 테스트 실행 (Red?)              │
│  2. 실패 분석                       │
│  3. 코드 수정 (Green)               │
│  4. 테스트 재실행                    │
│  5. 전부 통과? → Refactor → 완료    │
│     아직 실패? → 2로 돌아감         │
└─────────────────────────────────────┘
```

**각 반복에서:**
1. 테스트 실행 후 출력을 분석한다
2. 실패한 테스트의 원인을 파악한다
3. **가장 단순한 수정**으로 테스트를 통과시킨다
4. 수정이 다른 테스트를 깨뜨리지 않는지 확인한다
5. 모든 테스트 통과 시 리팩터링 기회를 확인한다

### 3. 결과 보고

루프 종료 후 결과를 요약한다:

```
=== Test Harness 결과 ===
반복 횟수: 4
최초 실패: 7건
최종 통과: 전체 통과 (15/15)
수정한 파일: 3개
  - core/engine.py (2곳)
  - services/api.py (1곳)
  - utils/helper.py (1곳)
```

### 4. SDD 스펙 검증 연동 (선택)

`specs/features/` 에 관련 스펙이 있으면:
- 테스트 통과 후 스펙의 검증 기준(AC-*)도 함께 확인한다
- 스펙 드리프트가 있으면 경고한다

## 핵심 원칙

- **최소 수정**: 한 번에 하나의 실패만 고친다
- **부작용 방지**: 수정이 다른 테스트를 깨뜨리면 즉시 롤백
- **투명성**: 매 반복마다 무엇을 수정했는지 로그
- **안전 장치**: max-iterations 도달 시 중단하고 진행 상황 보고

## 검증 기준

- [ ] 테스트 프레임워크 자동 감지
- [ ] 실패 테스트 분석 및 코드 수정
- [ ] 모든 테스트 통과 또는 max-iterations 도달
- [ ] 수정 내역 요약 보고
