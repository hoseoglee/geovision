# Triframe Skill

복잡한 작업을 3개의 독립적 관점(Generator, Reviewer, Researcher)으로 분리하여 품질을 높인다.

## 워크플로우

### 1. 요청 파악
- 사용자의 요청을 분석하여 Triframe 패턴이 적합한지 판단한다.
- 단순 작업(파일 하나 수정, 단답형 질문 등)이면 일반 방식으로 처리하고, Triframe을 적용하지 않는다.
- 요청의 핵심 목표와 제약조건을 정리한다.

### 2. Generator (생성자) 에이전트
- Agent 도구를 사용하여 서브에이전트로 실행한다.
- 역할: 사용자 요청에 대한 **초안(draft)**을 생성한다.
- 창의적이고 포괄적인 결과물을 우선한다.
- 출력: 초안 결과물 + 자신의 접근 방식 설명

### 3. Reviewer (검토자) 에이전트
- Agent 도구를 사용하여 서브에이전트로 실행한다.
- 역할: Generator의 초안을 **비판적으로 검토**한다.
- 검토 관점:
  - 정확성: 사실 오류, 논리적 결함
  - 완전성: 빠진 부분, 미흡한 부분
  - 일관성: 내부 모순, 스타일 불일치
  - 실용성: 실제 적용 가능성
- 출력: 문제점 목록 + 구체적 개선 제안

### 4. Researcher (연구자) 에이전트
- Agent 도구를 사용하여 서브에이전트로 실행한다.
- 역할: Reviewer의 지적사항을 바탕으로 **추가 조사/근거 수집**을 수행한다.
- 코드베이스 탐색, 기존 패턴 분석, 관련 문서 확인 등을 통해 개선에 필요한 근거를 마련한다.
- 출력: 조사 결과 + 근거 자료 + 권장 사항

### 5. 종합 및 최종 결과물 생성
- Generator의 초안, Reviewer의 검토, Researcher의 조사 결과를 종합한다.
- Reviewer가 지적한 문제점을 Researcher의 근거를 바탕으로 해결한다.
- 최종 결과물을 생성한다.

### 6. 결과 보고
- 사용자에게 각 단계별 결과를 요약 보고한다:
  ```
  ## Triframe 결과 요약

  ### Generator (생성)
  - [초안 핵심 내용 요약]

  ### Reviewer (검토)
  - [발견된 문제점 수]
  - [주요 개선점]

  ### Researcher (연구)
  - [조사 결과 요약]
  - [채택된 근거]

  ### 최종 결과
  - [종합 결과물]
  ```

## 핵심 원칙

- **독립성**: 각 에이전트는 자신의 역할에만 집중한다. Generator는 생성, Reviewer는 비판, Researcher는 조사.
- **병렬 가능 시 병렬**: Reviewer와 Researcher가 독립적으로 실행 가능한 경우 병렬로 실행한다. 단, Reviewer는 Generator 결과가 필요하고, Researcher는 Reviewer 결과가 필요하므로 기본적으로 순차 실행한다.
- **투명성**: 각 에이전트의 결과를 사용자에게 명확히 보여준다.
- **과도한 적용 지양**: Triframe은 복잡한 작업에만 적용한다. 오버헤드가 크므로 단순 작업에는 사용하지 않는다.

## 검증 기준

- [ ] Generator, Reviewer, Researcher 3개 서브에이전트가 실행됨
- [ ] Reviewer의 피드백이 Generator 결과에 반영됨
- [ ] 최종 산출물이 초기 버전보다 품질 향상됨
