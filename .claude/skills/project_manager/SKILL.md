# Project Manager Skill

프로젝트를 **하이레벨**에서 점검한다. 코드 한 줄이 아니라 프로젝트 전체를 놓고 "이 방향이 맞는가?"를 묻는 스킬.

## 사용법

```
/project_manager [project_path]        # 전체 점검 (6가지 관점)
/project_manager --focus <관점>         # 특정 관점만 점검
/project_manager --compare <before> <after>  # 리팩토링 전후 비교
```

**관점**: `목표`, `경계`, `데이터`, `확장`, `운영`, `이해`

## 검증 기준

- [ ] 6가지 관점 모두 평가되었는가 (또는 --focus 시 해당 관점)
- [ ] 각 관점에 구체적 근거(파일, 구조, 문서)가 포함되었는가
- [ ] "현재 상태"와 "권장 방향"이 구분되어 제시되었는가
- [ ] 우선순위가 매겨진 액션 아이템이 도출되었는가
- [ ] 결과가 papyrus에 저장되었는가 (프로젝트 문서)

## 워크플로우

### 1. 컨텍스트 수집

프로젝트를 이해하기 위한 정보를 수집한다.

```
병렬 Agent:
├─ Agent A: 프로젝트 구조 탐색
│   - 디렉토리 트리, 파일 수, 언어/스택
│   - 주요 설정 파일 (package.json, pyproject.toml, docker-compose.yml)
│   - README, 아키텍처 문서 읽기
│
├─ Agent B: 코드베이스 개요
│   - 모듈/패키지 목록, 각 모듈의 역할
│   - 의존성 그래프 (import 관계)
│   - 진입점(entrypoint) 식별
│
└─ Agent C: 프로젝트 이력
    - git log 최근 50커밋 — 변경 빈도, 핫스팟 파일
    - papyrus 아키텍처 문서 / 요구사항
    - 최근 보고서에서 프로젝트 관련 이슈
```

### 2. 6가지 관점 평가

수집된 정보를 기반으로 각 관점을 평가한다.

#### 관점 1: 목표 유효성 — "이 프로젝트가 왜 존재하는가?"

- 아키텍처 문서/README에 명시된 목표가 있는가
- 그 목표가 현재 코드와 일치하는가
- 목표가 바뀌었는데 구조는 그대로인 부분이 있는가
- 불필요해진 기능/모듈이 있는가

**평가 기준**:
| 상태 | 의미 |
|------|------|
| ✅ | 목표와 구현이 일치 |
| ⚠️ | 목표가 모호하거나 문서화 안 됨 |
| 🔴 | 목표와 구현이 괴리됨 |

#### 관점 2: 경계 설계 — "각 부분의 역할이 명확한가?"

- 하나의 모듈이 너무 많은 역할을 하지 않는가
- 분리해야 할 것이 합쳐져 있지 않은가
- 합쳐야 할 것이 불필요하게 나뉘어 있지 않은가
- 외부 시스템과의 경계가 명확한가 (API, DB, 메시지)

**신호**:
- 한 파일에 3개 이상의 관심사 → 분리 필요
- 2개 모듈이 항상 함께 수정됨 → 합치기 고려
- 순환 의존 → 경계 재설계

#### 관점 2.5: 중복 정리 — "같은 역할을 하는 것이 여러 곳에 있는가?"

코드가 아닌 **기능/역할/책임** 수준에서의 중복을 점검한다.

**점검 질문**:

1. **역할 중복** — 같은 일을 하는 모듈/서비스/시스템이 2개 이상 있는가?
   - 예: 보고를 저장하는 경로가 WebSocket과 HTTP 두 곳에 별도로 존재
   - 예: 알림을 보내는 시스템이 Worker와 anima-worker 양쪽에 존재

2. **경로 중복** — 같은 결과에 도달하는 경로가 여러 개인가?
   - 한 곳을 고쳤는데 다른 경로는 안 고쳐서 동작이 달라지는 문제
   - 어떤 경로가 "정답"인지 사용자/개발자가 혼동

3. **책임 분산** — 하나의 관심사가 여러 계층에 흩어져 있는가?
   - 예: 프로젝트 매칭 로직이 서버(태그 감지)와 Worker(Claude 추론) 양쪽에 존재
   - 하나로 모아야 할지, 의도적 분리인지 판단

4. **시스템 간 중복** — 독립 시스템들이 같은 기능을 각자 구현하고 있는가?
   - 예: Hermes Worker와 anima-worker의 역할이 겹치는 부분
   - 통합할지, 역할을 명확히 나눌지 판단

**정리 원칙**:

- **진짜 중복 vs 의도적 분리**: 변경 시 양쪽 다 수정해야 하면 진짜 중복. 독립적으로 변할 수 있으면 유지
- **단일 책임 경로**: 하나의 기능에는 하나의 정답 경로만. 나머지는 그 경로를 호출
- **통합 vs 역할 분리**: 겹치는 시스템은 "합칠 것인가, 경계를 명확히 할 것인가" 둘 중 하나를 결정
- **중복 제거의 비용**: 통합의 복잡도가 중복 유지의 비용보다 크면 건드리지 않음

#### 관점 3: 데이터 흐름 — "데이터가 어디서 오고 어디로 가는가?"

- 데이터의 진실 원천(SSOT)이 어디인가
- 같은 데이터가 여러 곳에 중복 저장되지 않는가
- 데이터 변환이 몇 단계를 거치는가 (많을수록 복잡)
- 상태가 어디에 저장되는가 (DB, 메모리, 파일)

**다이어그램 생성**:
```
[입력] → [처리 A] → [저장소] → [처리 B] → [출력]
```

#### 관점 4: 확장 방향 — "다음 기능이 자연스럽게 들어가는가?"

- 아키텍처 문서/백로그의 다음 기능을 추가하려면 어디를 수정해야 하는가
- 한 기능 추가에 몇 개 파일을 건드려야 하는가
- 플러그인/확장 포인트가 있는가
- 기술 부채가 확장을 막고 있는가

**테스트**: 백로그에서 3개 기능을 골라 "이걸 추가하려면?"을 시뮬레이션

#### 관점 5: 운영 가능성 — "문제가 생기면 찾을 수 있는가?"

- 로깅이 충분한가 (에러, 주요 이벤트)
- 모니터링/알림이 있는가
- 배포/롤백이 쉬운가
- 한 부분의 장애가 전체를 멈추게 하는가
- 백업/복구 전략이 있는가

**체크리스트**:
- [ ] 구조화된 로깅
- [ ] 헬스체크 엔드포인트
- [ ] 환경별 설정 분리
- [ ] 에러 알림
- [ ] 데이터 백업

#### 관점 6: 이해 가능성 — "새 사람이 30분 안에 파악할 수 있는가?"

- README가 최신인가
- 아키텍처 문서가 현실과 일치하는가
- 암묵적 규칙이 몇 개인가 (문서화 안 된 컨벤션)
- 모듈 이름만으로 역할을 유추할 수 있는가
- 진입점에서 전체 흐름을 따라갈 수 있는가

### 3. 종합 평가 + 액션 아이템

```
📋 프로젝트 점검 결과: {project_name}

| 관점 | 상태 | 핵심 발견 |
|------|------|----------|
| 목표 유효성 | ✅/⚠️/🔴 | {한줄 요약} |
| 경계 설계 | ✅/⚠️/🔴 | {한줄 요약} |
| 중복 정리 | ✅/⚠️/🔴 | {한줄 요약} |
| 데이터 흐름 | ✅/⚠️/🔴 | {한줄 요약} |
| 확장 방향 | ✅/⚠️/🔴 | {한줄 요약} |
| 운영 가능성 | ✅/⚠️/🔴 | {한줄 요약} |
| 이해 가능성 | ✅/⚠️/🔴 | {한줄 요약} |

전체 건강도: {7개 중 ✅ 비율}%

🔴 즉시 조치
1. {구체적 액션} — 이유: {근거}

⚠️ 권장 개선
2. {구체적 액션} — 이유: {근거}

✅ 강점 (유지)
- {잘 되어있는 점}

## 개선 제안

### 🔴 {관점명}: {문제 한줄}
- **현재**: {현상}
- **영향**: {방치 시 리스크}
- **방향**: {해결 방향}
- **효과**: {개선 후 기대}
- **규모**: 소/중/대

### ⚠️ {관점명}: {문제 한줄}
- **현재**: {현상}
- **방향**: {개선 방향}
- **안 하면**: {리스크}
```

### 4. 개선 제안

각 🔴/⚠️ 항목에 대해 구체적인 개선 방향을 제시한다.

**개선 제안 형식**:
```
### 개선 제안: {관점명}

**현재 상태**: {무엇이 문제인가}
**영향**: {이대로 두면 어떤 일이 벌어지는가}
**권장 방향**: {어떻게 바꿔야 하는가}
**예상 효과**: {바꾸면 무엇이 좋아지는가}
**연관 관점**: {이 변경이 다른 관점에 미치는 영향}
```

- 🔴 항목: 구체적 해결 방향 + 예상 작업 규모(소/중/대) 제시
- ⚠️ 항목: 개선 방향 + "지금 안 하면 어떻게 되는가" 리스크 명시
- ✅ 항목: 유지 포인트 + "이것이 흐트러지는 신호" 경고 조건

### 5. 결과 저장

- papyrus 파일 네이밍 규칙을 따른다:
  - 파일명 형식: `yyyy-mm-dd-hh-mm_프로젝트점검.md` (예: `2026-03-16-00-30_프로젝트점검.md`)
  - `~/papyrus/projects/{project}/`에 저장 (프로젝트 폴더 내)
  - 작업명은 한글로 간결하게
- YAML frontmatter 필수: `tags: [review]`, `date`, `project`
- 아키텍처 문서(`~/papyrus/projects/{project}/아키텍처.md`)의 `## 관련 보고서` 섹션에 옵시디언 링크 추가

## 제약 조건

- **코드 수정을 하지 않는다** — 방향을 제시할 뿐, 직접 고치지 않음
- **코드 레벨 개선은 developer improve** — 이 스킬은 구조/방향성만
- **스킬 시스템 점검은 anima health** — 프로젝트와 스킬 관리는 별개
- **주관적 판단을 명시한다** — "이것이 문제입니다"가 아니라 "이 관점에서 보면 이런 리스크가 있습니다"

## 핵심 원칙

- **한 문장 테스트**: "지금 구조가 앞으로 하고 싶은 일을 방해하는가?" — 방해하면 리팩토링, 아니면 유지
- **근거 기반**: 모든 평가에 구체적 파일/구조/데이터를 근거로 제시
- **실용주의**: "깨끗하게 만들고 싶다"는 이유만으로 리팩토링을 권하지 않음
- **궁극 목표 연결**: 프로젝트의 궁극 목표와 현재 구조의 갭을 식별

## 스킬 의존성

- **입력**: 아키텍처 문서 (papyrus), git log, 코드베이스
- **출력**: 점검 보고서 (papyrus), 액션 아이템
- **연계**: developer improve (코드 레벨 개선 실행), reflection (작업 결과 검토)
