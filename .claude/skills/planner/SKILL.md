# Planner 스킬 (기획자)

프로젝트의 아키텍처·요구사항·현재 상태를 분석하여 구현이 필요한 기능을 기획하는 스킬.

## 사용법

```
/기획자 <프로젝트명>                    # 해당 프로젝트의 기획 카드 생성
/기획자 <프로젝트명> --focus "영역"      # 특정 영역에 집중하여 기획
/기획자 <프로젝트명> --request "요청사항" # 사용자 요청 기반 상세 기획
```

## 핵심 원칙

1. **프로젝트 취지 우선**: 스펙 문서와 아키텍처 문서의 `[!goal] 궁극적 목표`를 반드시 먼저 읽고, 모든 기획이 이 취지에 부합하는지 확인
2. **Dedup-First**: 새 기능 기획 전 반드시 기존 기능과 중복 여부를 검토. 기존 기능 확장이 나은지, 새로 만드는 것이 나은지 판단
3. **과거 교훈 반영**: `~/.anima/soul/memory/lessons.md`에서 `process`, `understanding` 카테고리의 교훈을 확인하고 같은 실수 방지
4. **개발 패턴 활용**: `~/.anima/soul/memory/developer_patterns.md`에서 효과적이었던 전략을 기획에 반영
5. **최근 작업 흐름 반영**: `~/papyrus/records/` 최근 보고서를 참고하여 현재 프로젝트 방향성과 일관된 기획

## 실행 절차

### 1단계: 컨텍스트 수집
- `~/papyrus/projects/{프로젝트}/아키텍처.md` 읽기 (개요, 기술스택, 요구사항)
- `~/papyrus/projects/{프로젝트}/스펙.md` 읽기 (프로젝트 취지·목표)
- `~/papyrus/projects/{프로젝트}/_knowledge.md` 읽기 (프로젝트 지식)
- `~/.anima/soul/memory/lessons.md` — process, understanding, effective 교훈 확인
- `~/.anima/soul/memory/developer_patterns.md` — 개발 패턴/함정 확인
- 최근 보고서 3개의 작업 결과 섹션 참조

### 2단계: 분석 및 기획
- 기존 요구사항과 중복되지 않는 새로운 기능 식별
- 기존 기능 확장 vs 신규 기능 판단 (Dedup-First)
- 각 기획 항목에 대해:
  - **목적**: 왜 필요한지
  - **기대효과**: 어떤 가치를 주는지
  - **범위**: 포함/제외 사항
  - **수용 기준**: 완료 조건

### 3단계: 결과 출력
- 최대 5개 기획 카드 생성
- 각 카드는 구체적이고 실행 가능한 수준

### 4단계: 피드백
- 아키텍처.md 요구사항 섹션에 기획 카드 추가
- 교훈에서 적용한 항목에 `적용 확인` 표시

## 출력 형식

### 독립 사용 시
마크다운으로 기획 결과를 직접 표시하고 사용자와 논의.

### 칸반 파이프라인 사용 시
반드시 아래 JSON 형식으로만 출력:
```json
{"cards": [{"title": "카드 제목", "desc": "마크다운 설명"}, ...]}
```
최대 5개.

desc 템플릿:
```markdown
## 목적
이 기능이 왜 필요한지, 어떤 문제를 해결하는지

## 기대효과
구현 시 사용자/시스템에 어떤 가치를 주는지

## 범위
무엇을 포함하고, 무엇은 포함하지 않는지

## 수용 기준
완료로 판단할 수 있는 구체적 조건
```

## 검증 기준
- [ ] 프로젝트 아키텍처와 스펙을 먼저 읽었는가
- [ ] 기존 요구사항/기능과 중복 여부를 확인했는가
- [ ] 과거 교훈(process, understanding)을 참조했는가
- [ ] 각 기획 카드에 목적/기대효과/범위/수용기준이 모두 포함되었는가
- [ ] 궁극적 목표에 부합하는 기획인가
